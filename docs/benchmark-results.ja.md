# ChIP-Atlas Pipeline v2: ベンチマーク結果

## 1. 概要

本ドキュメントは、ChIP-Atlas Pipeline v2 のベンチマーク結果をまとめたものです。v2 では旧世代のバイオインフォマティクスツールを最新版に置き換え、1サンプルあたりの処理時間を約1日から約15-20分へと大幅に短縮しつつ、結果の品質を維持しています。

### テストしたパイプライン

| パイプライン | 説明 |
|----------|-------------|
| **Option A "Fast Classic"** | v1と同じ処理ステップ（QCなし、トリミングなし、単一ピークコーラー）で、ツールのみ最新版に置き換え（bwa-mem2, MACS3, samtools最新版） |
| **Option B "Modern"** | fastp によるQC/トリミングを追加し、bedtools genomecov + bedGraphToBigWig の代わりに deeptools bamCoverage を使用 |

各オプションは2つのモードでテストしました:

| モード | アライナー | 備考 |
|------|---------|-------|
| **CPU** | bwa-mem2 | デフォルトパイプライン |
| **GPU** | Parabricks fq2bam | GPU アクセラレーションによるアライメント＋ソート＋重複除去 |

### ベンチマーク対象ゲノム

| ゲノム | サイズ | テストサンプル数 |
|--------|------|----------------|
| sacCer3 | ~12 MB | 1（初期テスト） |
| ce11 | ~100 MB | 46（フルベンチマーク） |
| hg38 | ~3 GB | 18（フルベンチマーク） |

### ツールバージョン

| ツール | バージョン |
|------|---------|
| bwa-mem2 | 2.2.1 |
| Parabricks | 4.3.1 |
| MACS3 | 3.0.4 |
| samtools | 1.19.2 |
| bedtools | 2.31.1 |
| fastp | latest (Option B) |
| UCSC tools | 482 |

## 3. ce11 ベンチマーク結果

### ベンチマーク範囲

46 ce11 サンプル（6実験タイプ x 3リードティア x 約3サンプル）、PacBioサンプル1件を除外。

### テストした3つのパイプライン (Option A)

| パイプライン | ワークフロー | 説明 |
|----------|----------|-------------|
| CPU (default) | `option-a.cwl` | bwa-mem2、MACS3モデル構築あり |
| CPU (nomodel) | `option-a-nomodel.cwl` | bwa-mem2、MACS3 `--nomodel --extsize 200` |
| GPU (Parabricks) | `option-a-parabricks.cwl` | Parabricks fq2bam、MACS3 `--nomodel` |

### 成功率

| パイプライン | 成功 | 失敗 | 備考 |
|----------|-----|--------|-------|
| CPU (default) | 34 | 12 | MACS3モデル構築失敗6件、SE命名4件、sortグロブ1件、PacBio 1件 |
| CPU (nomodel) | 45 | 1 | PacBioサンプルのみ失敗 |
| GPU (Parabricks) | 45 | 0 | SE修正（`--in-se-fq`）後すべて成功 |

**結論**: `--nomodel --extsize 200` により、MACS3のモデル構築失敗をすべて解消でき、精度の低下もありません。

### 処理時間（パイプラインのみ、ダウンロード除く）

| リードティア | サンプル数 | CPU (nomodel) | GPU (Parabricks) | 高速化 |
|-----------|---------|---------------|-------------------|---------|
| Low (<10M) | 15 | 7分 | 5分 | 1.25倍 |
| Medium (10-50M) | 15 | 16分 | 11分 | 1.47倍 |
| High (>50M) | 15 | 43分 | 31分 | 1.37倍 |
| **全体** | **45** | **22分** | **16分** | **1.37倍** |

v1の約1日/サンプルと比較すると、v2パイプラインはce11において**CPU で約65倍**、**GPU で約90倍**高速です。

### ピーク数比較

**nomodel vs defaultモデル（CPU, q 1e-05）**:
- ATAC-Seq, DNase-seq, Histone, TFs: **ピーク数完全一致**（差異0）
- RNA polymerase: **2%未満の差異**（推定 vs 固定フラグメントサイズによる軽微な差）
- `--nomodel` をデフォルトとして安全に使用可能

**CPU vs GPU (q 1e-05)**:
- 45サンプルを比較
- **総ピーク数差異 0.9%**（171,677 vs 173,185）
- ほとんどのサンプルで+/-1-15ピークの差
- bwa-mem2 と Parabricks (BWA-MEM) はほぼ同一の結果を生成

## 4. hg38 ベンチマーク結果

### ベンチマーク範囲

18 hg38 サンプル（実験タイプ x リードティアごとに1件）、CPUとGPUパイプラインを並列実行。すべて Option A "Fast Classic" + `--nomodel` を使用。

### 処理時間

| リードティア | サンプル数 | CPU (nomodel) | GPU (Parabricks) | 高速化 |
|-----------|---------|---------------|-------------------|---------|
| Low (<10M) | 6 | 7分 | 5分 | 1.4倍 |
| Medium (10-50M) | 6 | 60分 | 31分 | 1.9倍 |
| High (>50M) | 6 | 118分 | 72分 | 1.6倍 |
| **全体** | **18** | **61分** | **36分** | **1.7倍** |

GPUによる高速化はhg38（1.7倍）の方がce11（1.37倍）より大きく、GPUアクセラレーションはゲノムサイズが大きいほど効果的です。

主な外れ値:
- Bisulfite-Seq 45Mリード: CPU 91分 vs GPU **8分**（10.2倍の高速化）
- Bisulfite-Seq 314Mリード: CPU 214分 vs GPU **67分**（3.1倍の高速化）
- 一部のサンプルではCPUがGPUより速い結果（ATAC-Seq 59M, Histone 123M）-- 両ベンチマーク並列実行によるGPU/CPUリソース競合が原因と考えられる

### ピーク数比較（CPU vs GPU, q 1e-05）

- 14サンプルを比較（GPU出力1件欠損）
- **総ピーク数差異 0.8%**（CPU 168,493 vs GPU 167,069）
- ほとんどのサンプルで差異1%未満、ce11の結果と一貫
- 外れ値1件: SRX26159220 (TFs) で約1,500ピーク（7%）の差異

### スケーリング比較: ce11 vs hg38

| 指標 | ce11 (100MBゲノム) | hg38 (3GBゲノム) | 比率 |
|--------|--------------------|--------------------|-------|
| CPU平均処理時間 | 22分 | 61分 | 2.8倍 |
| GPU平均処理時間 | 16分 | 36分 | 2.3倍 |
| GPU高速化 | 1.37倍 | 1.7倍 | ゲノムが大きいほどGPUの効果大 |

## 5. 2x2 比較: Option A vs B x CPU vs GPU (ce11)

4つのパイプラインバリアントすべてを46 ce11サンプルでベンチマーク。

### パイプライン処理時間（平均、パイプラインステップのみ）

|  | トリミングなし (Option A) | fastp トリミング (Option B) |
|--|----------------------|-------------------------|
| **CPU (bwa-mem2)** | 22分 | 17分 |
| **GPU (Parabricks)** | 16分 | **13分** |

### リードティア別

| ティア | A CPU | A GPU | B CPU | B GPU |
|------|-------|-------|-------|-------|
| Low (<10M) | 7m | 5m | 6m | 4m |
| Medium (10-50M) | 16m | 11m | 12m | 9m |
| High (>50M) | 43m | 31m | 33m | 27m |

### 成功率

| パイプライン | 成功 | 失敗 | 備考 |
|----------|-----|--------|-------|
| Option A CPU | 45 | 1 | PacBioサンプル失敗 |
| Option A GPU | 45 | 0 | |
| Option B CPU | **46** | **0** | fastp が PacBio リードをフィルタリングし、パイプライン成功 |
| Option B GPU | **46** | **0** | |

### 主な知見

1. **Option B は Option A より高速**（CPU: 17分 vs 22分、GPU: 13分 vs 16分）-- fastp によるリード数削減と、deeptools bamCoverage が bedtools genomecov + bedGraphToBigWig より高速であるため
2. **GPU はいずれのオプションでも約1.3倍の高速化**を実現
3. **Option B + GPU が最速**で平均13分 -- Option A CPU の1.7倍
4. **Option B はロバスト性も向上** -- fastp が品質フィルタとして機能し、PacBio リードを100%除去してパイプライン失敗を防止
5. **ピーク数は4バリアント間でほぼ同等** -- トリミングによる差異は5%未満
6. v1の約1日/サンプルと比較すると、最も遅いバリアント（Option A CPU, 22分）でも**約65倍高速**

### 推奨

GPU利用可能な環境では **Option B + GPU** を本番推奨。CPU のみのクラスタでは **Option B CPU** を推奨。Option A は推奨しません -- Option B の方が高速かつロバストで、デメリットがありません。

### ce11（v1ピークありの35サンプル + v1=0の10サンプル）

| 指標 | 値 |
|--------|-------|
| v2で回収されたv1ピーク | 平均 **約90%** |
| v2でより多くのピークを検出 | サンプルの77%（総ピーク数1.6倍） |
| v1=0だがv2でピーク検出 | 10サンプル（各数千ピーク） |

**実験タイプ別オーバーラップ:**

| タイプ | 平均オーバーラップ | 備考 |

| ATAC-Seq | 88% | 高リードサンプルでv2が2-40倍多くピーク検出 |
| TFs and others | 87% | 良好な一致、類似のカウント |
| DNase-seq | 87% | v2でより多く検出 |

### hg38（v1ピークありの12サンプル）

| 指標 | 値 |

| v2で回収されたv1ピーク | 平均 **約77%** |

| オーバーラップ <70% のサンプル | 2/12（下記外れ値参照） |

**サンプル別オーバーラップ:**

|------|---------|---------|---------|-------|
| ATAC-Seq (8M) | 3,316 | 10,657 | 98% | v2が3倍多く検出 |
| DNase-seq (50M) | 35,958 | 35,885 | 93% | ほぼ同一 |
| DNase-seq (72M) | 51,473 | 55,556 | 95% | v2がやや多い |

**外れ値:**
- ~~SRX25595131 (Histone, 10Mリード, SE): v1=8,797ピーク、v2=201ピーク~~ **解決済み**: この実験には **2つのSRRラン**（SRR30125615: 3Mリード + SRR30125616: 7Mリード）があるが、ベンチマークでは最初のランのみダウンロード（データの30%）。両ランをダウンロード・結合後: **v2=6,633ピーク** -- v1の8,797と整合（ツール変更による約25%の差異であり、バグではない）。**修正**: `download-experiment.sh` で実験あたりの全ランを解決し、FASTQを結合するよう対応。v1の挙動と一致。
- SRX25254554 (TFs, 10Mリード): v1=28,075、v2=20,239（オーバーラップ66%）。v2のピーク数が少ない。

### マルチ閾値比較（q 1e-05, 1e-10, 1e-20）

**v1 vs v2 CPU ピーク数比率（全閾値）:**

| ゲノム | q05 (v2/v1) | q10 (v2/v1) | q20 (v2/v1) |
|--------|------------|------------|------------|
| ce11 (35サンプル) | 1.5x | 1.5x | 1.7x |
| hg38 (11サンプル、外れ値除く) | 1.0x | 0.9x | 0.8x |

- ce11: v2は**全閾値でより多くのピークを検出**し、厳しいカットオフほど比率が増加 -- v2の追加ピークは高信頼度
- hg38: q05では**v2とv1がほぼ同等**、厳しい閾値ではv2がやや少ない
- ゲノム間の差異は、異なるゲノム構造（ce11 ~100MB vs hg38 ~3GB）に対するアライナー固有の挙動を反映している可能性

**v2 CPU vs GPU 閾値間の一貫性:**

| 閾値 | ce11 (45サンプル) | hg38 (18サンプル) |
|-----------|------------------|-------------------|
| q05 | 0.8%の差異 | 0.8%の差異 |
| q10 | 1.1%の差異 | -- |
| q20 | 1.3%の差異 | -- |

CPUとGPUは全閾値でほぼ同一の結果を生成しており、アライナーの選択（bwa-mem2 vs Parabricks BWA-MEM）がピークコーリングへの影響が最小限であることを確認しました。

### 解釈

1. **v2はv1のピークの大部分を回収**（ce11で約90%、hg38で約77%）-- コアシグナルは保存されている
2. **v2はce11でより多くのピークを検出**（1.5倍）するが、**hg38ではほぼ同等** -- v1-v2の差異はゲノム依存
3. **v1=0のサンプルがv2でピーク検出** -- リグレッションではなく改善（ce11で10サンプル）
4. **一部のサンプルでv2のピーク数が減少** -- ツールの違いによる想定内の結果。ユーザーはq値閾値で調整可能
5. **オーバーラップは実験タイプにより異なる** -- RNA polymerase が最も高い一致度、ATAC-Seq で最も多くの新規ピーク
6. **CPUとGPUは互換性あり** -- 全閾値で1.5%未満のピーク差異

## 8. 発見された問題と修正

### sacCer3 初期テスト（2026-03-20）

| 問題 | 修正 |
|-------|-----|
| `samtools markdup` に `fixmate -m` が必要 | name-sort -> fixmate -> coord-sort フローを追加 |
| CWL `float` 型が微小q値（1e-10, 1e-20）を0に丸める | `string` 型に変更 |
| 複数のBiocontainers Dockerイメージタグが存在しない | 全タグを検証・修正 |
| Parabricks がリードグループに `PU` と `LB` フィールドを要求 | fq2bam ツールに追加 |

### ce11 ベンチマーク（2026-03-25）

| 問題 | 修正 |
|-------|-----|
| MACS3 モデル構築が低シグナルサンプルで失敗 | `--nomodel --extsize 200` を追加 |
| Single-end FASTQ命名の不一致（`SRR.fastq` vs `SRR_1.fastq`） | スクリプトで柔軟なFASTQ検出 |
| Parabricks SEリードに `--in-fq` ではなく `--in-se-fq` が必要 | CWLツールで条件分岐引数 |
| ENA の PacBio データ `_subreads.fastq` 命名 | 柔軟なFASTQ検出 + PacBioをバリデーションから除外 |
| CWL `float` が小さなq値を丸める | `string` 型に変更 |
| ピーク検出なし時に MACS3 `xls` 出力が欠損 | 出力をオプション化（`File?`） |

### hg38 ベンチマーク（2026-03-27）

| 問題 | 修正 |
|-------|-----|
| SRX25595131 外れ値: マルチラン実験で最初のランのみダウンロード | `download-experiment.sh` で実験あたりの全ランを解決しFASTQを結合 |

### データ品質に関する観察

SRX2170085 (ce11, Bisulfite-Seq) は、SRAメタデータで誤ってラベルされた **PacBio RS II** サンプルです。ショートリードインデックスに対するマッピング率は4.5%。**対応**: v2のサンプル選択にインストルメントフィルタを追加し、PacBio/ONTサンプルを除外。

注: ダウンロード時間は含まれていません。NIGの高帯域ネットワーク（100 Gbps InfiniBand + NII/SINETによる高速外部接続）により、ベンチマークマシンと比較してダウンロードのオーバーヘッドは大幅に削減される見込みです。

### スケーリング参考: 200K hg38 サンプルの処理時間

| シナリオ | サンプルあたり | 200K hg38サンプル | ダウンロード込み |

| CPUクラスタ（10ノード） | 61分 | ~2.3年 | + ダウンロード時間 |

**結論**: 大規模処理には大幅な並列化が必要です。現実的な再処理スケジュールには、100以上のCPUノードまたは10以上のGPUノードを持つクラスタが必要です。推奨デプロイ先はNIGスーパーコンピュータで、78 CPUノードで約50-70日（40万サンプル）です。

## 初期テスト結果（sacCer3）

sacCer3 での最初のテストラン（SRX22049197, H3K4me3, 約950Kリード）:

| 閾値 | v1 (MACS2 + Bowtie2) | v2 bwa-mem2 | v2 Parabricks |
|-----------|---------------------|-------------|---------------|
| q 1e-05   | 429                 | 539         | 539           |
| q 1e-10   | --                  | 396         | 397           |
