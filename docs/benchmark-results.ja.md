# ChIP-Atlas Pipeline v2: ベンチマーク結果

## 概要

ChIP-Atlas Pipeline v2は、旧世代のツールを最新版に置き換えるアップグレードプロジェクトである。遺伝研スパコン（専用AMD EPYCノード6台、各128コア）上で2つの実装をベンチマークした:

- **Option B CWL** — モジュラーCWLワークフロー、各ツールが中間ファイルを介して個別ステップとして実行
- **Option B Fast** — 最適化シングルパスパイプライン、パイプ処理により中間ファイル不要

hg38検証サンプル54件（289リード〜3.14億リード）での主要結果:

| 指標 | v1パイプライン | Option B CWL（16コア） | Option B Fast（32コア） |
|------|-------------|----------------------|------------------------|
| 平均総処理時間（10-50Mリード） | 約1日 | 57分 | **20分** |
| 平均総処理時間（50-100Mリード） | 約1日 | 93分 | **38分** |
| サンプルあたりピークディスク使用量 | 不明 | 約60 GB | **約18 GB** |
| 3億リード以上のサンプル | 未テスト | クラッシュ（ディスクquota超過） | **約146分** |

## 3. 遺伝研ベンチマーク: Option B Fast（パイプスルー）

3つの最適化を実装したシングルパスパイプライン:

1. **パイプスルー**: `fastp | bwa-mem2 | samtools sort | fixmate | sort | markdup` — 中間ファイルをディスクに書き込まない
2. **並列実行**: markdup後にbamCoverageとMACS3を同時実行
3. **MACS3単一実行**: q=1e-05で1回実行し、awkフィルターで1e-10と1e-20を抽出（3回の個別実行を置換）

ジョブあたり32コア、Apptainerコンテナ、NVMeスクラッチ。完了30/54サンプルのデータ。

## 5. ダウンロード性能

`fast-download.sh`によるソース認識型ルーティング:
- SRR/ERR → ENAミラー（aria2c、8並列接続）
- DRR → DDBJ（aria2c、キャッシュ済みfastqlist）
- フォールバック → Apptainer経由fasterq-dump

SRX→SRRアクセッション変換: TogoID一括API（togoid.dbcls.jp）。

### リード数階層別ダウンロード時間（遺伝研）

| リード数階層 | 平均DL時間 | 備考 |
|------------|----------|------|
| <1M | 12秒 | 無視できるレベル |
| 1-10M | 30秒 | 高速 |
| 10-50M | 4-9分 | ソースミラーに依存 |
| 50-100M | 3-9分 | ENA遅延により変動 |
| 100M+ | 3-16分 | 大容量ファイル、ネットワーク依存 |

ダウンロード時間は中規模サンプルの総処理時間の約10%、ネットワークが遅い場合は最大約25%。

| ゲノム | v2でのv1ピーク再現率 | v2がより多くのピークを検出 |

| hg38（12サンプル） | 約77% | q=1e-05でほぼ同等 |

## 8. 主要な設計判断と修正

| 判断 | 理由 |
|------|------|
| 常に`format=BAM`を使用（BAMPEではなく） | BAMPEは適切にペアリングされたフラグメントが必要；誤ラベルPEデータで失敗 |
| 全MACS3に`--nomodel --extsize 200` | 低シグナルサンプルでのモデル構築失敗を排除 |
| バックグラウンド/インプットコントロールなし | ChIP-Atlasポリシー — 40万件超のサンプルに対する均一処理 |
| MACS3単一実行 + awkフィルター | q=1e-05で1回実行 + 1e-10/1e-20のフィルター、3回の個別実行を置換 |
| SRX→SRR変換にTogoID | NCBI e-utilsは同時アクセスでレート制限；TogoIDは一括変換対応 |
| udockerよりApptainer | ネイティブカーネルサポート、~/.udockerキャッシュ不要（約900 GB節約） |
| 中間ファイルにNVMe /data1 | Lustreより3倍高速なI/O、大規模実行時の競合を軽減 |
