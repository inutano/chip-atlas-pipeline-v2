# ChIP-Atlas Pipeline v2: 進捗ログ

開発、ベンチマーク、および検証作業の時系列記録。

## 2026-03-20: パイプライン初期実装とテスト

**完了項目:**

1. **CWLツール定義**（11ツール）:
   - `fasterq-dump.cwl` — SRAダウンロード (sra-tools 3.0.10)
   - `bwa-mem2-align.cwl` — アライメント (bwa-mem2 2.2.1)
   - `samtools-sort.cwl` — name/coordinateソート (samtools 1.19.2)
   - `samtools-fixmate.cwl` — markdup用のmate scoreタグ追加
   - `samtools-markdup.cwl` — 重複除去（非推奨のrmdupを置換）
   - `samtools-mapped-count.cwl` — RPM正規化用のリード数カウント
   - `bedtools-genomecov.cwl` — BedGraphカバレッジ (bedtools 2.31.1)
   - `bedgraphtobigwig.cwl` — BedGraph → BigWig (UCSC tools 482)
   - `macs3-callpeak.cwl` — コントロールなしのピークコール (MACS3 3.0.4)
   - `bedtobigbed.cwl` — BED → BigBed (UCSC tools 482)
   - `parabricks-fq2bam.cwl` — GPU高速化アライメント+ソート+重複除去 (Parabricks 4.3.1)

2. **2つのワークフローバリアント**:
   - `option-a.cwl` — CPUパイプライン（bwa-mem2 → sort → fixmate → sort → markdup → ...）
   - `option-a-parabricks.cwl` — GPUパイプライン（fq2bamがalign+sort+markdupを置換）

3. **検証用サンプルセット**: 6ゲノム × 6実験タイプ × 3リード数階層で301サンプルを選定

4. **初回テスト実行** sacCer3 (SRX22049197, H3K4me3, 約95万リード):

| 閾値 | v1 (MACS2 + Bowtie2) | v2 bwa-mem2 | v2 Parabricks |
|------|---------------------|-------------|---------------|
| q 1e-05 | 429             | 539         | 539           |
| q 1e-10 | —               | 396         | 397           |
| q 1e-20 | —               | 286         | 286           |

- bwa-mem2とParabricksはほぼ同一の結果（±1ピーク）
- v2はq 1e-05でv1より約25%多くのピークを検出（bwa-mem2がBowtie2より高感度なため想定通り）

**テスト中に発見・修正した問題:**
- `samtools markdup`は事前に`fixmate -m`が必要 → name-sort → fixmate → coord-sortフローを追加
- CWLの`float`型が極小のq値（1e-10、1e-20）を0に丸める → `string`型に変更
- 複数のBiocontainers Dockerイメージタグが存在しない → 全タグを検証・修正
- Parabricksはリードグループに`PU`と`LB`フィールドが必須 → fq2bamツールに追加

## 2026-03-25: ce11フルベンチマーク完了（Option A「Fast Classic」）

**以下のベンチマークはすべてOption A「Fast Classic」** — v1と同じ処理ステップ（QCなし、トリミングなし、全タイプ共通のピークコーラー）で、ツールのみ最新版に置換。Option B「Modern」（fastp、実験タイプ別コーラー）は未テスト。

**ベンチマーク範囲**: ce11サンプル46件（6実験タイプ × 3リード数階層 × 約3サンプル）、PacBioサンプル1件は除外。

### テストした3つのパイプライン

| パイプライン | ワークフロー | 説明 |
|------------|-------------|------|
| CPU（デフォルト） | `option-a.cwl` | bwa-mem2、MACS3モデル構築あり |
| CPU（nomodel） | `option-a-nomodel.cwl` | bwa-mem2、MACS3 `--nomodel --extsize 200` |
| GPU（Parabricks） | `option-a-parabricks.cwl` | Parabricks fq2bam、MACS3 `--nomodel` |

### 成功率

| パイプライン | 成功 | 失敗 | 備考 |
|------------|------|------|------|
| CPU（デフォルト） | 34 | 12 | MACS3モデル構築失敗6件、SE命名4件、ソートglob 1件、PacBio 1件 |
| CPU（nomodel） | 45 | 1 | PacBioサンプルのみ失敗 |
| GPU（Parabricks） | 45 | 0 | SE修正（`--in-se-fq`）後すべて成功 |

**結論**: `--nomodel --extsize 200`はMACS3のモデル構築失敗をすべて解消し、精度への影響はない。

### 処理時間（パイプラインのみ、ダウンロード除く）

| リード数階層 | サンプル数 | CPU（nomodel） | GPU（Parabricks） | 高速化率 |
|------------|-----------|---------------|-------------------|---------|
| Low（<1000万） | 15 | 7分 | 5分 | 1.25倍 |
| Medium（1000万-5000万） | 15 | 16分 | 11分 | 1.47倍 |
| High（>5000万） | 15 | 43分 | 31分 | 1.37倍 |
| **全体** | **45** | **22分** | **16分** | **1.37倍** |

v1の約1日/サンプルと比較して、v2パイプラインはCPUで**約65倍**、GPUで**約90倍**高速（ce11データ）。

### ピーク数比較

**nomodel vs デフォルトモデル（CPU、q 1e-05）**:
- ATAC-Seq、DNase-seq、Histone、TFs: **ピーク数完全一致**（差異0）
- RNA polymerase: **2%未満の差異**（推定vs固定フラグメントサイズによる軽微な差）
- `--nomodel`をデフォルトとして安全に使用可能

**CPU vs GPU（q 1e-05）**:
- 45サンプルを比較
- **総ピーク数差異0.9%**（171,677 vs 173,185）
- ほとんどのサンプルで±1-15ピークの差異
- bwa-mem2とParabricks（BWA-MEM）は実質的に同一の結果

### ダウンロード速度

| 方法 | 速度 | 備考 |
|------|------|------|
| fasterq-dump（NCBI） | ベースライン | シングルスレッドSRA変換 |
| aria2c + ENA | **2.5倍高速** | 8並列HTTP接続、生成済みFASTQ |
| aria2c + DDBJ | DRRで利用可能 | キャッシュ済みfastqlistでパス検索 |

アクセッション番号プレフィックスによるダウンロードルーティング: DRR → DDBJ → ENA → fasterq-dump、SRR/ERR → ENA → fasterq-dump。

### 発見・修正した問題

| 問題 | 修正 |
|------|------|
| MACS3モデル構築が低シグナルサンプルで失敗 | `--nomodel --extsize 200`を追加 |
| シングルエンドFASTQ命名（`SRR.fastq` vs `SRR_1.fastq`） | スクリプトで柔軟なFASTQ検出 |
| ParabricksのSEリードに`--in-se-fq`が必要 | CWLツールで条件分岐 |
| ENA `_subreads.fastq`命名（PacBioデータ） | 柔軟なFASTQ検出 + PacBioを検証から除外 |
| CWL `float`型が極小q値を切り捨て | `string`型に変更 |
| MACS3 `xls`出力がピークなし時に欠落 | 出力をオプション化（`File?`） |

### データ品質に関する発見

- SRX2170085（ce11、Bisulfite-Seq）は**PacBio RS II**サンプルで、SRAメタデータでの分類が不適切。ショートリードインデックスに対するマッピング率は4.5%。v1パイプラインでもインストゥルメントモデルによるフィルタリングが必要だった。**推奨事項**: v2のサンプル選定にインストゥルメントフィルタを追加し、PacBio/ONTサンプルを除外する。

## 2026-03-27: hg38ベンチマーク完了（Option A「Fast Classic」）

**ベンチマーク範囲**: hg38サンプル18件（すべてOption A「Fast Classic」+ `--nomodel`を使用）（実験タイプ × リード数階層ごとに1件）、CPUとGPUパイプラインを並列実行。

### 処理時間

| リード数階層 | サンプル数 | CPU（nomodel） | GPU（Parabricks） | 高速化率 |
|------------|-----------|---------------|-------------------|---------|
| Low（<1000万） | 6 | 7分 | 5分 | 1.4倍 |
| Medium（1000万-5000万） | 6 | 60分 | 31分 | 1.9倍 |
| High（>5000万） | 6 | 118分 | 72分 | 1.6倍 |
| **全体** | **18** | **61分** | **36分** | **1.7倍** |

hg38でのGPU高速化率（1.7倍）はce11（1.37倍）より大きい — GPU加速は大きなゲノムでより効果的。

特筆すべき結果:
- Bisulfite-Seq 4500万リード: CPU 91分 vs GPU **8分**（10.2倍高速化）
- Bisulfite-Seq 3.14億リード: CPU 214分 vs GPU **67分**（3.1倍高速化）
- 一部のサンプルでCPUがGPUより高速（ATAC-Seq 5900万、Histone 1.23億） — 両ベンチマーク並列実行によるリソース競合の可能性

### ピーク数比較（CPU vs GPU、q 1e-05）

- 14サンプルを比較（GPU出力1件欠損）
- **総ピーク数差異0.8%**（168,493 CPU vs 167,069 GPU）
- ほとんどのサンプルで1%未満の差異、ce11の結果と一致
- 外れ値1件: SRX26159220（TFs）で約1,500ピーク差（7%） — 要調査

### ChIP-Atlas全データ再処理の所要時間見積もり（hg38）

パイプライン平均処理時間に基づく見積もり（ダウンロード除く）:

| シナリオ | サンプルあたり | hg38 20万サンプル | 備考 |
|---------|-------------|------------------|------|
| CPU（1ノード、8コア） | 61分 | 約23年 | 非現実的 |
| GPU（1ノード、1 GPU） | 36分 | 約14年 | 非現実的 |

### マルチ閾値比較（q 1e-05、1e-10、1e-20）

**v1 vs v2 CPUピーク数比率（全閾値）:**

| ゲノム | q05 (v2/v1) | q10 (v2/v1) | q20 (v2/v1) |
|--------|------------|------------|------------|
| ce11（35サンプル） | 1.5倍 | 1.5倍 | 1.7倍 |
| hg38（11サンプル、外れ値除外） | 1.0倍 | 0.9倍 | 0.8倍 |

- ce11: v2は**全閾値で一貫してより多くのピークを検出**、厳しい閾値ほど比率が増加 — v2の追加ピークは高信頼度
- hg38: v2とv1はq05で**ほぼ同等**、厳しい閾値ではv2がやや少ない
- ゲノム間の差異はアライナーのゲノム構造への依存性を反映している可能性（ce11 ~100MB vs hg38 ~3GB）

**v2 CPU vs GPU 閾値間の一貫性:**

| 閾値 | ce11（45サンプル） | hg38（18サンプル） |
|------|-------------------|-------------------|
| q05 | 0.8%差 | 0.8%差 |
| q10 | 1.1%差 | — |
| q20 | 1.3%差 | — |

CPUとGPUは全閾値でほぼ同一の結果 — アライナーの選択（bwa-mem2 vs Parabricks BWA-MEM）がピークコールへの影響は最小限。

### 解釈

1. **v2はv1ピークの大部分を回収**（ce11で約90%、hg38で約77%） — コアシグナルは保存
2. **v2はce11でより多くのピークを検出**（1.5倍）だが、**hg38ではほぼ同等** — v1-v2差異はゲノム依存
3. **v1=0のサンプルがv2でピーク検出** — これは退行ではなく改善（ce11で10サンプル）
4. **一部サンプルでv2のピークが少ない** — ツール差異により想定内；q値閾値で調整可能
5. **オーバーラップは実験タイプにより変動** — RNA polymeraseが最も高い一致、ATAC-Seqで最も新規ピークが多い
6. **CPUとGPUは互換性あり** — 全閾値で1.5%未満のピーク差異

## 2026-03-28: 2x2ベンチマーク完了（ce11）

4つのパイプラインバリアントすべてを46 ce11サンプルでベンチマーク。

### パイプライン処理時間（平均、パイプラインステップのみ）

|  | トリミングなし（Option A） | fastpトリミング（Option B） |
|--|--------------------------|---------------------------|
| **CPU（bwa-mem2）** | 22分 | 17分 |
| **GPU（Parabricks）** | 16分 | **13分** |

### リード数階層別

| 階層 | A CPU | A GPU | B CPU | B GPU |
|------|-------|-------|-------|-------|
| Low（<1000万） | 7分 | 5分 | 6分 | 4分 |

### 成功率

| パイプライン | 成功 | 失敗 | 備考 |
|------------|------|------|------|
| Option A CPU | 45 | 1 | PacBioサンプル失敗 |
| Option A GPU | 45 | 0 | |
| Option B CPU | **46** | **0** | fastpがPacBioリードを100%フィルタ → パイプライン成功 |
| Option B GPU | **46** | **0** | |

### 主要な知見

1. **Option BはOption Aより高速**（CPU: 17分 vs 22分、GPU: 13分 vs 16分） — fastpでリード数が若干減少し、deeptools bamCoverageがbedtools genomecov + bedGraphToBigWigより高速
2. **GPUはさらに約1.3倍の高速化**
3. **Option B + GPUが最速**（13分平均） — Option A CPUの1.7倍高速
4. **Option Bはより堅牢** — fastpが品質ゲートとして機能、PacBioリードを100%フィルタし失敗を防止
5. **ピーク数は4バリアント間で類似**（<5%差異）
6. v1の約1日/サンプルと比較して、最も遅いバリアント（Option A CPU、22分）でも**約65倍高速**

### 推奨

**Option B + GPU**を本番運用に推奨（GPU利用可能な場合）。**Option B CPU**をCPUのみのクラスターに推奨。Option Aは非推奨 — Option Bの方が高速かつ堅牢で、デメリットなし。
