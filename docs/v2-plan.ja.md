# ChIP-Atlas Pipeline v2: アップグレード計画

## 設計思想

オリジナルのChIP-Atlasパイプラインは意図的にシンプルな設計となっている。各サンプルの実験条件（抗体、細胞種、プロトコル、シーケンス深度）が異なるため、サンプルごとの最適化は大規模処理では現実的ではない。v1パイプラインはすべてのサンプルに対して同一の基本処理を均一に適用しており、このシンプルさは制約ではなく設計上の特長である。

v2では、速度と近代化のトレードオフを評価するため、**2つのパイプラインオプション**を提案する。

### Option A: "Fast Classic" — 同じステップ、高速なツール

v1の処理ロジックをそのまま維持し（同じステップ、可能な限り同じパラメータ）、各ツールを高速な最新版に置き換える。新しいステップは追加しない。

**メリット:**
- v1との結果の比較が最も容易 — 継続性の検証がしやすい
- 新たなバイアス導入のリスクが最小限
- 元の設計思想（均一な基本処理）を尊重
- 実装・保守がシンプル

**デメリット:**
- 品質向上の可能性を見送る（例：アダプタートリミング）
- 実験タイプ別のベストプラクティスを活用できない
- v1データに存在する品質上の問題がそのまま残る

### Option B: "Modern" — ステップの更新とベストプラクティスの適用

パイプライン全体を近代化：QC/トリミングの追加、実験タイプ別のピークコーラー使用、現在のベストプラクティスの適用。

**メリット:**
- サンプルごとの結果品質が向上
- 実験タイプに応じた処理（ChIP-seq vs ATAC-seq vs CUT&Tag）
- 現在のコミュニティ標準に準拠
- CUT&Tagなどの新しい実験タイプに対応しやすい

**デメリット:**
- v1との結果の差異が大きくなる — 継続性の検証が困難
- 実験タイプごとのワークフロー分岐による複雑性の増加
- サンプル間比較に影響するバイアス導入のリスク
- チューニング・保守すべきパラメータが増加

### 評価戦略

1. Option AとOption Bを別々のCWLワークフローとして実装
2. 同一の代表的サンプルセットで両方を実行
3. 結果を比較：A vs v1、B vs v1、A vs B
4. 本番運用でどちらを採用するか決定（または用途に応じて両方使用）

## 目標

1. **処理時間の大幅短縮**（現在 約1日/サンプル）
2. **ツールの近代化** — 10年前のバージョンを最新に置き換え
3. **GPU高速化** — 有効な箇所で活用
4. **CWLベースのオーケストレーション** — ポータブル、再現可能、ランナー非依存
5. **カスタムCWLランナー** — 最小限、最速、あらゆるインフラに柔軟対応
6. **全範囲** — 一次処理と二次解析の両方を書き直し
7. **新しい実験タイプのサポート** — ChIP-seq/DNase-seq/ATAC-seq/Bisulfite-seqに加えCUT&Tag
8. **v1との検証** — ピーク数とオーバーラップを比較し差異を把握

## インフラストラクチャ

- **主要環境**: オンプレミスGPUワークステーションおよびHPCクラスター
- **GPU**: RTX（旧世代）、DGX Spark（H200）、より高性能なGPUマシンも利用可能
- **RAM**: クラスターノードあたり128〜256 GB（ワークステーションはそれより少ない）
- **クラウド**: 必要に応じて利用可能（AWS/GCP）
- **コンテナ**: Singularity/Apptainer
- **ハードウェア構成間のベンチマーク** — 異なるGPUモデル、メモリサイズでの比較

## Phase 1: ベンチマークとツール選定

### 1.1 ~~v1パイプラインのプロファイリング~~（スキップ）

v1の挙動を再現することは現実的ではない — 元のシェルスクリプトでは多くの変数やファイルパスがハードコードまたは暗黙的に宣言されている。代わりに以下の方針とする：

- 新しいパイプライン（Option AおよびB）を直接構築
- 新パイプラインでステップごとの処理時間をベンチマーク
- chip-atlas.dbcls.jpに既に公開されているv1の既存結果と出力を比較
- 新パイプラインはいずれにせよ高速になるため、v1のプロファイリングは労力に見合わない

### 1.2 アライナー評価

同一サンプルセットで3つのアライナーを実行し、速度・メモリ・アライメント品質を比較：

| アライナー | タイプ | 備考 |
|-----------|--------|------|
| bwa-mem2 | CPU (AVX-512) | 最速のCPUアライナー、BWAのドロップイン置換 |
| minimap2 | CPU | 汎用性が高く高速、ロングリード対応で将来性あり |
| Parabricks (GPU BWA-MEM) | GPU (NVIDIA) | 極めて高速、商用ライセンスが必要 |

比較指標：
- サンプルあたりの実時間
- ピークメモリ使用量
- v1のBowtie2ベースラインとのマッピング率比較
- 下流のピークコール結果の一貫性

### 1.3 ツールマッピング: Option A vs Option B

| ステップ | v1ツール | Option A (Fast Classic) | Option B (Modern) |
|---------|---------|------------------------|-------------------|
| SRAダウンロード | SRA Toolkit 2.3.2-4 | SRA Toolkit最新版 (fasterq-dump) | Aと同じ |
| FASTQ QC | (なし) | (なし) | fastp (QC+トリミングを一括処理) |
| トリミング | (なし) | (なし) | fastp |
| アライメント | Bowtie2 2.2.2 | bwa-mem2 / minimap2 / Parabricks | Aと同じ |
| BAM処理 | SAMtools 0.1.19 | SAMtools最新版 (1.20+) | Aと同じ |
| 重複除去 | samtools rmdup | samtools markdup | Aと同じ |
| カバレッジトラック | bedtools 2.17.0 + bedGraphToBigWig | bedtools最新版 + bedGraphToBigWig | deeptools bamCoverage (BAM→BigWig直接変換) |
| ピークコール | MACS2 2.1.0 (全タイプ共通) | MACS3 (全タイプ共通) | MACS3 (ChIP-seq), HMMRATAC (ATAC-seq), SEACR (CUT&Tag) |
| フォーマット変換 | UCSC bedToBigBed | UCSC tools最新版 | Aと同じ |

**主な違い**: Option Aはv1と同じステップを維持（QCなし、トリミングなし、全タイプ共通のピークコーラー）。Option BはQC/トリミングを追加し、実験タイプ別のピークコーラーを使用。

### 1.4 CUT&Tagに関する考慮事項（Option Bのみ）

- CUT&TagはChIP-seqよりバックグラウンドが低い — SEACRが推奨ピークコーラー
- CWLワークフロー内でピークコールの分岐が必要
- アライメントパラメータが異なる可能性あり（例：フラグメントサイズの期待値）

### 2.2 CWL設計原則

- **CWL v1.2**仕様
- ツールごとに1つの`CommandLineTool` — 粒度が細かく、再利用可能、テスト可能
- `Workflow`ドキュメントでツールをパイプラインに構成
- `scatter`によるサンプル間の並列実行
- すべてのツールをSingularityコンテナでラップ
- 入出力の型を厳密に定義しバリデーション
- 初期テストは**cwltool**で実施
- Option AとOption Bは同じツール定義を共有し、ワークフローレベルでのみ異なる

### 2.3 カスタムCWLランナー（サブプロジェクト）

- 最小限の実装 — このパイプラインに必要なCWL v1.2サブセットをサポート
- SLURM/SGE/ローカルへの直接ジョブ投入（中間レイヤーなし）
- Singularityネイティブ（Docker変換レイヤーなし）
- 依存関係解決付きの並列ステップ実行
- 大規模処理向け設計（40万サンプル以上）
- 言語は未定（Rust? Go? Python?）

## Phase 3: 二次解析の書き直し

### 3.1 ターゲット遺伝子解析

- シェル+bedtoolsをCWLワークフローに置き換え
- bedtools最新版でピーク-TSSオーバーラップを計算
- STRING連携をPythonで書き直し
- 出力: TSVテーブル

### 3.2 共局在解析

- カスタムJavaツール（`coloCA.jar`）をPython実装に置き換え
- 同じアルゴリズム: ガウス分布フィット → Zスコアグループ → ペアワイズスコアリング
- STRINGスコアを統合
- CWL CommandLineToolとしてラップ

### 3.3 エンリッチメント / In Silico ChIP

- PythonとCWLラッパーで書き直し
- bedtools intersect + scipyによるFisherの正確検定
- statsmodelsによるBH補正

## Phase 4: 検証

### 4.1 サンプル選定

- 代表的なサブセットを選定: ゲノムあたり10〜20サンプル
- 異なる実験タイプをカバー（ChIP-seq、DNase-seq、ATAC-seq）
- 高品質サンプルとボーダーラインのサンプルの両方を含める
- 様々な抗原クラスを含める（ヒストンマーク、転写因子、クロマチン制御因子）

### 4.2 三者間比較

| 比較 | 目的 |
|------|------|
| Option A vs v1 | ツールのアップグレードのみで結果が壊れないことを検証 |
| Option B vs v1 | QC/トリミング/タイプ別コーラー追加の影響を把握 |
| Option A vs Option B | 近代化ステップの効果を分離 |

指標：
- **ピーク数**: 各q値閾値でのピーク数
- **ピークオーバーラップ**: Jaccard係数とオーバーラップ係数
- **探索的分析**: 合否閾値を設定する前に差異を可視化
- オプション: BigWigシグナル相関（Pearson/Spearman）は後の段階で

### 4.3 検証ツール

- `validate-vs-v1.py`: 2つのBEDファイルを入力し、ピーク数・オーバーラップ統計を報告、比較プロットを生成
- テストスイートの一部として実行

## Phase 5: 本番デプロイとマイグレーション

### 5.1 判断ポイント

Phase 4の検証後に決定：
- **Option Aを採用** — v1との一貫性が最重要の場合
- **Option Bを採用** — 品質向上が差異を正当化する場合
- **両方を運用** — 用途に応じて使い分ける場合

### 5.2 段階的ロールアウト

1. 未処理の1万サンプル以上を選択したオプションで処理
2. 既存40万サンプルのサブセットを再処理して検証
3. 結果が満足できれば全面的に再処理

### 5.3 インフラ更新

- メタデータフィルタリング: Pythonで書き直し（シェルスクリプトを置換）
- 増分更新ロジック: 新しいSRAアクセッションを検出しキューに追加
- データ配信: 後方互換性のため同じURL構造を維持

## タイムライン（大まかなフェーズ、時間見積もりではない）

1. **ベンチマーク** — v1のプロファイリング、アライナーとツール候補の評価
2. **CWL開発** — Option AとOption Bのワークフロー構築
3. **カスタムランナー** — 最小限のCWLランナー開発（#2と並行可能）
4. **二次解析** — PythonとCWLで書き直し
5. **検証** — 三者間比較（A vs v1、B vs v1、A vs B）
6. **判断と本番運用** — オプション選択、残りのサンプルを処理
