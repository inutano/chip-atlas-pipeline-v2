## 設計思想

オリジナルのChIP-Atlasパイプラインは意図的にシンプルな設計となっている。各サンプルの実験条件（抗体、細胞種、プロトコル、シーケンス深度）が異なるため、サンプルごとの最適化は大規模処理では現実的ではない。v1パイプラインはすべてのサンプルに対して同一の基本処理を均一に適用しており、このシンプルさは制約ではなく設計上の特長である。

v2では、速度と近代化のトレードオフを評価するため、**2つのパイプラインオプション**を提案する。

### Option A: "Fast Classic" — 同じステップ、高速なツール

v1の処理ロジックをそのまま維持し（同じステップ、可能な限り同じパラメータ）、各ツールを高速な最新版に置き換える。新しいステップは追加しない。

**メリット:**
- v1との結果の比較が最も容易 — 継続性の検証がしやすい
- 新たなバイアス導入のリスクが最小限
- 元の設計思想（均一な基本処理）を尊重
- 実装・保守がシンプル

**デメリット:**
- 品質向上の可能性を見送る（例：アダプタートリミング）
- 実験タイプ別のベストプラクティスを活用できない
- v1データに存在する品質上の問題がそのまま残る

### Option B: "Modern" — ステップの更新とベストプラクティスの適用

パイプライン全体を近代化：QC/トリミングの追加、実験タイプ別のピークコーラー使用、現在のベストプラクティスの適用。

**メリット:**
- サンプルごとの結果品質が向上
- 実験タイプに応じた処理（ChIP-seq vs ATAC-seq vs CUT&Tag）
- 現在のコミュニティ標準に準拠
- CUT&Tagなどの新しい実験タイプに対応しやすい

**デメリット:**
- v1との結果の差異が大きくなる — 継続性の検証が困難
- 実験タイプごとのワークフロー分岐による複雑性の増加
- サンプル間比較に影響するバイアス導入のリスク
- チューニング・保守すべきパラメータが増加

### 評価戦略

1. Option AとOption Bを別々のCWLワークフローとして実装
2. 同一の代表的サンプルセットで両方を実行
3. 結果を比較：A vs v1、B vs v1、A vs B
4. 本番運用でどちらを採用するか決定（または用途に応じて両方使用）

## 重要ポリシー: ピークコールにおけるバックグラウンド/インプットコントロール不使用

ChIP-Atlasでは、ピークコールを**バックグラウンドデータ（インプットDNA / ネガティブコントロール）なし**で意図的に実行している。これは見落としではなく、設計上の意思決定である。

- **理由**: 公共リポジトリの40万件以上の実験を対象とする規模では、各ChIPサンプルに対応するインプットコントロールを確実に特定・ペアリングすることは現実的ではない。インプットデータが存在しない、ラベルが誤っている、またはSRAメタデータに明示的に記載されていないケースが多い。
- **仕組み**: MACS2/MACS3をコントロールBAMなしで実行する。代わりにローカルバックグラウンドモデルに対してピークをコールし、ユーザーは3つのq値閾値（1e-05、1e-10、1e-20）を用いて結果のストリンジェンシーをフィルタリングする。
- **これはChIP-Atlasの本質的な特徴** — コントロールの有無にかかわらず、すべての公共データを均一に処理することを可能にしている。
- **v2でもこのポリシーを維持する**: Option AもOption Bも、バックグラウンド/インプットコントロールなしでピークコールを行うこと。コントロールペアリングのロジックは実装しない。

## 目標

1. **処理時間の大幅短縮**（現在 約1日/サンプル）
2. **ツールの近代化** — 10年前のバージョンを最新に置き換え
3. **GPU高速化** — 有効な箇所で活用
4. **CWLベースのオーケストレーション** — ポータブル、再現可能、ランナー非依存
5. **カスタムCWLランナー** — 最小限、最速、あらゆるインフラに柔軟対応
6. **全範囲** — 一次処理と二次解析の両方を書き直し
7. **新しい実験タイプのサポート** — ChIP-seq/DNase-seq/ATAC-seq/Bisulfite-seqに加えCUT&Tag
8. **v1との検証** — ピーク数とオーバーラップを比較し差異を把握

### 1.1 ~~v1パイプラインのプロファイリング~~（スキップ）

v1の挙動を再現することは現実的ではない — 元のシェルスクリプトでは多くの変数やファイルパスがハードコードまたは暗黙的に宣言されている。代わりに以下の方針とする：

### 1.2 アライナー評価

同一サンプルセットで3つのアライナーを実行し、速度・メモリ・アライメント品質を比較：

| アライナー | タイプ | 備考 |
|-----------|--------|------|
| bwa-mem2 | CPU (AVX-512) | 最速のCPUアライナー、BWAのドロップイン置換 |
| minimap2 | CPU | 汎用性が高く高速、ロングリード対応で将来性あり |
| Parabricks (GPU BWA-MEM) | GPU (NVIDIA) | 極めて高速、商用ライセンスが必要 |

比較指標：
- サンプルあたりの実時間
- ピークメモリ使用量
- v1のBowtie2ベースラインとのマッピング率比較
- 下流のピークコール結果の一貫性

### 1.3 ツールマッピング: Option A vs Option B

| ステップ | v1ツール | Option A (Fast Classic) | Option B (Modern) |

### 1.4 CUT&Tagに関する考慮事項（Option Bのみ）

- CUT&TagはChIP-seqよりバックグラウンドが低い — SEACRが推奨ピークコーラー
- CWLワークフロー内でピークコールの分岐が必要
- アライメントパラメータが異なる可能性あり（例：フラグメントサイズの期待値）

## Phase 4: 検証 [x] Option A / [ ] Option B

### 4.1 サンプル選定

データソース: `https://chip-atlas.dbcls.jp/data/metadata/experimentList.tab`

#### 層別化の次元

1. **ゲノム**（現行の6アセンブリ、レガシーアセンブリはスキップ）:
   - hg38, mm10, rn6, dm6, ce11, sacCer3

2. **実験タイプ**（6種類）:
   - Histone, TFs and others, ATAC-Seq, DNase-seq, RNA polymerase, Bisulfite-Seq
   - スキップ: Input control, Unclassified, No description, Annotation tracks

3. **リード数階層**（カラム8の最初のカンマ区切り値に基づく）:
   - Low: 1000万リード未満
   - Medium: 1000万〜5000万リード
   - High: 5000万リード超

#### 選定方法

- **ゲノム × 実験タイプ × リード数階層**の各組み合わせについて:
  - アクセッション番号の降順でソート（新しい実験を優先 — 最新のシーケンス機器にバイアス）
  - 上位から**3サンプル**を選定
- 生物種ごとに複数サンプルを確保し、ゲノム構造に起因するマッパー/ピークコーラーの挙動の違いを捕捉
- **想定サイズ**: 最大約270サンプル（6ゲノム × 6タイプ × 3階層 × 3サンプル）、組み合わせが少ない箇所ではそれより少なくなる

#### 選定スクリプト

`scripts/select-validation-samples.py` — experimentList.tabを読み込み、フィルタリングと層別化を適用し、選定サンプルリストを`data/validation-samples.tsv`に出力

### 2026-03-27: v1 vs v2 ピークオーバーラップ解析（Option A「Fast Classic」）

chip-atlas.dbcls.jpからv1のBEDファイルをダウンロードし、bedtools intersectでv2 Option A結果とのピークオーバーラップを比較。

#### ce11（v1ピークあり35サンプル + v1=0が10サンプル）

| 指標 | 値 |
|------|-----|
| v1ピークのv2での回収率 | 平均**約90%** |
| v2がより多くのピークを検出 | 77%のサンプル（総ピーク数1.6倍） |
| v1=0だがv2でピーク検出 | 10サンプル（各数千ピーク） |

**実験タイプ別オーバーラップ:**

| タイプ | 平均オーバーラップ | 備考 |
|--------|-------------------|------|
| RNA polymerase | 97% | 非常に高い一致 |
| Histone | 93% | 良好、v2がより多く検出 |
| ATAC-Seq | 88% | 高リード数サンプルでv2が2-40倍多く検出 |
| TFs and others | 87% | 良好な一致、類似したピーク数 |
| DNase-seq | 87% | v2がより多く検出 |

#### hg38（v1ピークあり12サンプル）

| 指標 | 値 |
|------|-----|
| v1ピークのv2での回収率 | 平均**約77%** |
| 90%以上のオーバーラップ | 4/12サンプル（DNase-seq、ATAC-Seq） |

#### マルチ閾値比較（q 1e-05、1e-10、1e-20）

**v1 vs v2 CPUピーク数比率（全閾値）:**

| ゲノム | q05 (v2/v1) | q10 (v2/v1) | q20 (v2/v1) |
|--------|------------|------------|------------|
| ce11（35サンプル） | 1.5倍 | 1.5倍 | 1.7倍 |
| hg38（11サンプル、外れ値除外） | 1.0倍 | 0.9倍 | 0.8倍 |

- ce11: v2は**全閾値で一貫してより多くのピークを検出**、厳しい閾値ほど比率が増加 — v2の追加ピークは高信頼度
- hg38: v2とv1はq05で**ほぼ同等**、厳しい閾値ではv2がやや少ない
- ゲノム間の差異はアライナーのゲノム構造への依存性を反映している可能性（ce11 ~100MB vs hg38 ~3GB）

**v2 CPU vs GPU 閾値間の一貫性:**

| 閾値 | ce11（45サンプル） | hg38（18サンプル） |
|------|-------------------|-------------------|
| q05 | 0.8%差 | 0.8%差 |
| q10 | 1.1%差 | — |
| q20 | 1.3%差 | — |

CPUとGPUは全閾値でほぼ同一の結果 — アライナーの選択（bwa-mem2 vs Parabricks BWA-MEM）がピークコールへの影響は最小限。

#### 解釈

1. **v2はv1ピークの大部分を回収**（ce11で約90%、hg38で約77%） — コアシグナルは保存
2. **v2はce11でより多くのピークを検出**（1.5倍）だが、**hg38ではほぼ同等** — v1-v2差異はゲノム依存
3. **v1=0のサンプルがv2でピーク検出** — これは退行ではなく改善（ce11で10サンプル）
4. **一部サンプルでv2のピークが少ない** — ツール差異により想定内；q値閾値で調整可能
5. **オーバーラップは実験タイプにより変動** — RNA polymeraseが最も高い一致、ATAC-Seqで最も新規ピークが多い
6. **CPUとGPUは互換性あり** — 全閾値で1.5%未満のピーク差異

**外れ値（解決済）**: ~~SRX25595131（Histone、hg38、SE）でv1=8,797 → v2=201~~ この実験は**2つのSRRラン**（SRR30125615: 300万リード + SRR30125616: 700万リード）を持つが、ベンチマークでは最初のランのみダウンロード（データの30%）。両ラン結合後: **v2=6,633ピーク** — v1の8,797と整合（ツール差異による約25%の差、バグではない）。**修正**: `download-experiment.sh`で実験ごとに全ランを解決しFASTQを結合、v1の挙動と一致。
